## 一、项目定位：我们到底在做什么、给谁看

### 1.1 一句话定位

面向高校的 **SaaS 化就业大数据平台**：持续采集招聘网站职位数据 → 清洗入库 → 多维统计分析 → 自动生成就业分析报告（含 AI 智能摘要）→ 按学生画像个性化推送岗位 → 对外提供开放 API。

### 1.2 交付背景与约束（决定一切技术决策）

| 约束 | 内容 | 对设计的影响 |
|---|---|---|
| 时间 | 实训共 12 天，**实际编码窗口约 5 天**（第 7 天组队，第 11 天集中编码，第 12 天答辩） | 一切从"能演示、能讲清"出发，拒绝铺摊子 |
| 人力 | 4 名大二学生 | 每人负责边界清晰的 1 个纵向切片，互相只依赖接口 |
| 考核 | 需求分析占先启阶段 70%、架构+数据库占精化阶段 75%、编码占构建阶段 75%、演示占答辩 40% | **文档与代码并重**；演示流程必须完整流畅 |
| 技术要求 | SpringBoot、Redis、Nginx、Kafka、Spark（任务书列明） | 全部覆盖：前四项已落地，Spark 以"可插拔实现"方式覆盖（见 3.4） |
| 加分点 | 创新钻研、复杂工程、SpringAI/AI 工具应用（课程第 6 天专门教学） | 保留 2 个可落地的 AI 亮点：LLM 智能报告摘要 + 技能缺口诊断 |

### 1.3 四类用户与核心场景

| 角色 | 登录后看到什么 | 核心操作 |
|---|---|---|
| **管理员 ADMIN** | 数据大屏（5 维度图表）、采集任务管理、报告模板/记录、用户管理 | 配置采集任务→触发分析→生成报告 |
| **学生 STUDENT** | 个性化推荐流（带匹配分+缺失技能提示）、职位详情、收藏/投递、消息、报告查看 | 完善画像→浏览推荐→投递→收到推送 |
| **教师 TEACHER** | 本校学生就业意向列表、学生求职行为统计、教学建议（技能缺口） | 掌握学情→依据市场数据调整教学 |
| **企业 HR** | 职位发布管理、人才画像浏览（脱敏） | 发布职位→查看人才池 |

## 二、深度优化说明：相对初稿改了什么、为什么

初稿（memory-bank 中 v1.0 设计）技术雄心很大，但按 5 天编码窗口评估，**多个组件属于"装了跑不起来、跑起来讲不清"的高风险项**。本次优化原则：**每一项保留的技术都必须在答辩时既能演示、又能被任何一个组员讲明白**。

### 2.1 技术栈收敛（核心决策）

| 初稿方案 | 优化后 | 理由 |
|---|---|---|
| HDFS + Hive + HBase 三件套存储 | **MySQL 单库**（原始表 raw_job_data + 明细表 job_detail + 结果表 analysis_result 三层，保留数仓分层思想） | 万级数据量用 Hadoop 属于典型过度设计；Docker 起 Hadoop 全家桶需要 16G+ 内存，组员笔记本跑不动；分层思想在 MySQL 上同样能讲 |
| Spark on Yarn 集群跑分析 | **SQL 聚合 + Java 内存计算**（`AnalysisJobService`），接口保持可插拔——如需演示 Spark，把实现替换为 Spark local 模式即可，调用方零改动 | 分析逻辑不变，去掉集群运维负担；接口抽象本身就是答辩可讲的架构设计点 |
| XXL-Job 分布式调度 | **Spring @Scheduled**（默认），XXL-Job 改为可选开关（`xxl.job.enabled`，代码保留） | XXL-Job 需要单独部署 admin 服务端；@Scheduled 零成本且够用 |
| Neo4j 知识图谱 + ES + Milvus 向量库 + Three.js 3D 大屏 + Prophet 预测（6 大亮点） | **砍到 2 个可落地亮点**：① LLM 智能报告摘要（`AiSummaryService`，已有框架+降级方案）② 技能缺口诊断（教师端，纯 SQL/Java 即可实现） | 6 个亮点 = 6 个新中间件 = 0 个能完成；2 个亮点分别对应课程 SpringAI 教学点和"教学改革辅助"的项目本意，且都不引入新部署组件 |
| OAuth2 资源服务器做开放 API 鉴权 | **apiKey+apiSecret 换 Redis 临时 Token + 拦截器**（已实现） | OAuth2 配置复杂、排错困难；轻量方案 60 行代码实现同等演示效果，答辩时能画出完整时序图 |

**保留不动的**：SpringBoot 2.7 模块化单体、MyBatis-Plus 多租户插件、JWT 认证、Kafka 消息管道、WebMagic 爬虫、Redis 缓存、Vue 3 前端、Docker Compose、Nginx。这些已经落地且直接对应任务书的技术要求。

### 2.2 本轮已补齐的框架（相对初稿的增量）

| 类别 | 内容 |
|---|---|
| 基础设施修补 | ① common 补上缺失的 Redis / Spring Security 依赖；② JWT 白名单放行 `/api/open/**` 与 Knife4j 文档；③ 开启方法级权限（`@PreAuthorize` 按角色控制）；④ 新增 `UserContextHolder`（业务层获取当前登录人）；⑤ 新增统一分页响应 `PageResult<T>`；⑥ XXL-Job 改为条件装配（默认关闭，消除启动报错）；⑦ 修复 HealthControllerTest（初稿遗留的上下文加载失败） |
| **auth** | 用户管理接口（分页/新增/编辑/启停，ADMIN 专用）——支撑管理后台用户管理页 |
| **analysis** | ① 数据清洗链路：`JobDataCleanListener`（Kafka 双消费组）+ `DataCleanService`（去重/标准化/技能提取/非法丢弃）；② 统计引擎：`AnalysisJobService` 5 维度聚合（行业/城市/学历/技能/趋势）；③ `AnalysisScheduler` 定时调度（逐租户）；④ 手动重算接口 `POST /api/analysis/rebuild`；⑤ 职位写入口 `saveJob/removeJob/getJobById`（供 HR 端与开放 API） |
| **report** | 模板 CRUD + 报告生成引擎（六步流程已跑通：模板→数据→AI 摘要→Freemarker 渲染→导出→落盘记录）+ PDF/Word 导出器 + **LLM 智能摘要**（OpenAI 兼容接口 + 自动降级） |
| **recommend** | 学生画像 CRUD + **匹配算法**（技能 40/城市 25/薪资 20/学历 15 规则打分，含匹配理由与缺失技能分析）+ 推送服务 + 行为闭环（VIEW/FAVORITE/APPLY）+ 教师端/HR 端接口 + 每日 8 点推送调度 |
| **api** | Token 签发/校验（Redis）+ 限流拦截器（固定窗口计数）+ 开放数据接口（职位/大盘/技能热度/行业分布，带缓存） |

> 以上全部通过 `mvn clean compile` 与现有单元测试验证。**剩余待实现项**见第七章清单（主要是前端页面填充与若干 TODO 标注点）。

## 三、总体架构

### 3.1 分层架构图

```
│  展示层    occupation-web-ui (Vue3 + Element Plus + ECharts)    │

├────────────────────────────────────────────────────────────────┤

│            ApiTokenInterceptor + RateLimitInterceptor（开放API） │
├────────────────────────────────────────────────────────────────┤
│  业务层    SpringBoot 2.7 模块化单体（8 个 Maven Module）        │
│   ┌─────────┬─────────┬──────────┬─────────┬───────────┬─────┐ │
│   │  auth   │ crawler │ analysis │ report  │ recommend │ api │ │
│   │ 认证用户 │ 采集调度 │ 清洗+统计 │ 报告+AI │ 画像+推荐  │开放 │ │
│   └─────────┴─────────┴──────────┴─────────┴───────────┴─────┘ │
│                    common（响应/异常/多租户/Kafka/上下文）        │
├────────────────────────────────────────────────────────────────┤
│  中间件    Kafka（采集↔清洗解耦） Redis（缓存/Token/限流）        │
├────────────────────────────────────────────────────────────────┤
│  存储层    MySQL 8.0 —— 14 张表，三层数据：                      │
│            raw_job_data(原始层) → job_detail(明细层) → analysis_result(汇总层) │
├────────────────────────────────────────────────────────────────┤
│  外部服务  LLM API（DeepSeek/通义，可选，自动降级）               │
└────────────────────────────────────────────────────────────────┘
```

### 3.2 核心数据流（答辩必讲，建议画成时序图）

```
① 采集   CrawlerTask 配置 → CrawlerService 启动 WebMagic 爬虫（Mock/BOSS）
              → JobPipeline 封装 JobDataMessage → 发送 Kafka(raw-job-data)
② 双路消费  Kafka ─┬→ [data-cleaner-group] KafkaConsumerService → raw_job_data（原始归档）
                   └→ [job-clean-group]   JobDataCleanListener → DataCleanService
                        （去重/标准化/技能提取）→ job_detail（明细层）
③ 统计   AnalysisScheduler(每日2:00) / POST /api/analysis/rebuild(手动)
              → AnalysisJobService 5维度聚合 → analysis_result（汇总层，按租户）
④ 消费   Dashboard(管理端图表) ← AnalysisService ← analysis_result + Redis缓存
         报告 ← ReportGeneratorService（模板+数据+AI摘要→HTML/PDF/Word）
         推荐 ← JobMatchService（画像×job_detail 规则打分）→ 推送(push_record)
         开放API ← OpenDataService（聚合缓存）+ Token/限流拦截器
```

**为什么用 Kafka 双消费组**（答辩高频问题）：归档与清洗互不阻塞——清洗逻辑出 bug 不丢原始数据，可用补偿任务（每小时的 `cleanPendingRawData`）重放；这就是"解耦+削峰+可重放"三个消息队列价值点的现场证明。

### 3.3 模块依赖关系（严格单向，禁止反向依赖）

```
occupation-web（启动器，聚合一切）
 ├─ common ←──────────── 所有模块的地基
 ├─ auth → common
 ├─ crawler → common
 ├─ analysis → common
 ├─ report → common + analysis     （报告需要分析数据）
 ├─ recommend → common + analysis  （推荐需要职位数据）
 └─ api → common + analysis        （开放API需要职位数据）
```

**跨模块调用只走 Service 接口**（`AGENTS.md` 规则 5）。当前的三个跨模块契约：

| 接口 | 提供方 | 调用方 | 方法 |
|---|---|---|---|
| `AnalysisService` | analysis | report | `getDashboard(query)` |
| `JobDetailService` | analysis | recommend / api | `queryJobs(query)` / `getJobById(id)` / `saveJob(dto)` / `removeJob(id)` |
| `AiSummaryService` | report | （report 内部） | `summarize(dashboard)` |

### 3.4 关于 Spark（任务书技术点的处理方式）

`AnalysisJobService` 接口就是为 Spark 预留的插拔点。当前实现（`AnalysisJobServiceImpl`）用 SQL 聚合；如答辩需要展示 Spark，新建 `SparkAnalysisJobServiceImpl` 用 `SparkSession(local[*])` 通过 JDBC 读 job_detail → DataFrame groupBy → 写回 analysis_result，Spring `@Primary` 切换即可，**调用方（调度器/Controller）零改动**。这个"面向接口的可替换实现"本身就是架构设计的加分讲点。文档化建议：答辩 PPT 中画出该插拔点。

### 3.5 多租户 SaaS 设计（项目最大的架构亮点，务必讲透）

- **行级隔离**：业务表带 `tenant_id`，MyBatis-Plus `TenantLineInnerInterceptor` 自动在 SQL 注入条件，业务代码零侵入。
- **共享数据**：`job_detail`、`raw_job_data`、`sys_tenant` 三表配置为忽略表（全平台共享职位数据，各校共用，节约采集成本）。
- **链路**：登录签发 JWT（含 tenantId）→ 过滤器解析 → `TenantContextHolder`(ThreadLocal) → SQL 自动过滤 → 请求结束清理。
- **定时任务**：无请求上下文，调度器显式遍历租户逐一设置上下文（见 `AnalysisScheduler`）。

## 十、风险与应对

| 风险 | 概率 | 应对 |

| 真实爬虫被反爬/结构变化 | 高 | **演示一律走 Mock 采集器**（链路完全一致）；BOSS 采集器作为"已实现的扩展能力"讲解 |
