## 3. 版本控制策略

### 3.1 版本控制工具

- **工具**：Git
- **远程仓库**：GitHub（https://github.com/casuccielisa671-lang/abc.git）
- **本地仓库**：`e:\occupation`
- **主分支**：`main`

### 3.2 分支模型

| 分支类型 | 命名 | 用途 | 生命周期 |
|---|---|---|---|
| 主分支 | `main` | 稳定可发布代码 | 长期 |
| 开发分支 | `develop` | 日常开发集成 | 长期 |
| 功能分支 | `feature/<模块>-<描述>` | 单个功能开发 | 合并后删除 |
| 修复分支 | `hotfix/<描述>` | 线上问题紧急修复 | 合并后删除 |
| 发布分支 | `release/<版本号>` | 版本发布前测试 | 发布后删除 |

### 3.3 提交规范

提交信息格式：

```
<类型>: <简短描述>

<正文（可选）>
```

| 类型 | 含义 | 示例 |
|---|---|---|
| `feat` | 新功能 | `feat: P2 A组完成 AnalysisServiceImpl` |
| `fix` | 修复缺陷 | `fix: 修复职位查询薪资区间判断错误` |
| `docs` | 文档更新 | `docs: 更新先启阶段配置管理计划` |
| `refactor` | 代码重构 | `refactor: 抽取 Dashboard 查询通用方法` |
| `test` | 测试相关 | `test: 增加 JobDetailService 单元测试` |
| `chore` | 构建/工具 | `chore: 升级 MyBatis-Plus 版本` |

### 3.4 标签管理

| 标签类型 | 命名 | 时机 |
|---|---|---|
| 里程碑标签 | `milestone-<编号>` | 里程碑完成时 |
| 版本标签 | `v<主>.<次>.<修订>` | 发布时 |
| 阶段标签 | `phase-<阶段名>` | 阶段结束时 |

示例：

```bash
git tag -a v1.0.0 -m "职业能力大数据服务平台 1.0.0 正式发布"
git tag -a milestone-2 -m "精化阶段设计评审通过"
```

## 4. 变更管理流程

### 4.1 变更类型

| 类型 | 说明 | 审批人 |
|---|---|---|
| 需求变更 | 功能范围、业务流程变化 | 项目经理 + 业务代表 |
| 设计变更 | 架构、数据库、接口变化 | 架构师 |
| 代码变更 | 实现层面的 bug 修复或重构 | 组长 |
| 环境变更 | 配置、依赖、部署变化 | 运维工程师 |

### 4.2 变更流程

1. **提出变更**：变更申请人在 issue/变更单中描述变更原因、影响范围、预期结果。
2. **影响评估**：架构师或组长评估变更对进度、成本、质量的影响。
3. **审批决策**：按变更类型由对应审批人决定接受/拒绝/延期。
4. **实施变更**：在 feature 分支上实施，遵循代码评审与测试流程。
5. **验证变更**：测试人员验证变更效果，更新测试用例。
6. **关闭变更**：记录变更结果，归档相关文档。

## 5. 环境管理

### 5.1 环境分类

| 环境 | 用途 | 访问权限 | 部署方式 |
|---|---|---|---|
| 开发环境 | 本地开发调试 | 开发团队 | Docker Compose |
| 测试环境 | 集成测试、系统测试 | 开发/测试 | Docker Compose / 测试服务器 |
| 预发环境 | 发布前验证 | 运维/测试 | 与生产一致 |
| 生产环境 | 正式对外服务 | 运维 | 待定 |

### 5.2 环境配置隔离

- 数据库连接、Redis 地址、Kafka 地址等环境相关配置放入 `application.yml` 或 `.env`。
- 敏感信息（密码、API Key、Token）使用环境变量或配置中心管理，禁止硬编码。
- 不同环境使用独立的 Spring Boot Profile：`application-dev.yml`、`application-test.yml`、`application-prod.yml`。

### 5.3 环境一致性保障

- 使用 `docker-compose.yml` 统一编排 MySQL、Redis、Kafka、Nginx 服务。
- 所有开发人员本地使用同一套 `docker-compose.yml`，避免环境差异。
- 关键依赖版本在 `pom.xml` 和 `package.json` 中锁定。

## 6. 构建与发布管理

### 6.1 构建规范

- 统一使用 Maven 构建后端：`mvn clean compile` 本地编译，`mvn clean package -DskipTests` 打包。
- 统一使用 npm 构建前端：`npm install && npm run build`。
- 构建产物禁止提交到 Git 仓库。

### 6.2 发布流程

1. 从 `develop` 分支创建 `release/<版本号>` 分支。
2. 在 release 分支上进行回归测试和缺陷修复。
3. 测试通过后合并到 `main` 分支并打标签 `v<版本号>`。
4. 将标签版本部署到生产环境。
5. 保留发布记录，包括版本说明、变更清单、已知问题。

### 6.3 回滚机制

- 保留最近 3 个版本的构建产物。
- 线上出现问题时，优先通过 Git 标签回滚到上一稳定版本。
- 数据库变更需单独编写回滚脚本，与升级脚本一一对应。

## 8. 配置审计

### 8.1 审计内容

- 配置项是否齐全、命名是否规范。
- 提交信息是否符合规范。
- 分支是否及时合并和清理。
- 环境配置是否外置、敏感信息是否脱敏。

### 8.2 审计频率

- 每周五由组长进行小组内配置审计。
- 每个阶段末由配置管理员进行项目级配置审计。
