## 1. 引言

### 1.1 编写目的

本文档面向项目经理、架构师、开发工程师、测试工程师，明确职业能力大数据服务平台（OCP）的功能需求、非功能需求、业务规则与验收标准，作为后续设计、开发、测试的共同依据。

### 1.2 项目背景

高校人才培养与就业市场需求之间存在信息不对称。学校难以及时掌握行业用人趋势，学生难以精准匹配适合自己的岗位，教师缺乏数据支撑进行教学改革。OCP 旨在通过大数据采集、分析、报告生成与智能推荐，打通人才供给侧与需求侧的信息壁垒。

### 1.3 术语与缩略语

## 2. 总体描述

### 2.1 系统目标

搭建一套集数据采集、分析、报告、推荐、开放接口于一体的职业能力大数据服务平台，服务于四类用户：

- **管理员**：管理用户、租户、采集任务、报告模板、系统配置。
- **教师**：查看所指导学生画像、就业行为、获取班级/专业就业分析。
- **学生**：完善个人画像、浏览岗位、收藏/投递、接收推荐与推送。
- **HR（企业）**：发布/管理职位、浏览人才、处理投递。

### 2.2 用户角色

| 角色 | 主要职责 | 使用端 |
|---|---|---|
| 管理员 | 系统配置、数据监控、报告管理 | 管理后台 |
| 教师 | 学生管理、就业指导、数据分析 | 教师端 |
| 学生 | 画像维护、职位浏览、投递 | 学生端 |
| HR | 职位管理、人才浏览、投递处理 | HR 端 |
| 第三方系统 | 调用开放 API 获取数据 | API 客户端 |

### 2.3 系统边界

**OCP 内部包含**：

- 数据采集与调度系统
- 数据清洗与存储系统
- 大数据分析与计算系统
- 报告生成与导出系统
- 岗位推荐与推送系统
- 统一认证与多租户系统
- 管理后台、学生端、教师端、HR 端
- 对外开放 API 与文档

**OCP 外部依赖**：

- 招聘网站（BOSS 直聘、智联招聘等）
- 政府公开数据源
- 第三方短信/邮件服务商（P5 后可选）
- 学校教务系统（P5 后可选）

### 2.4 约束条件

- 开发周期：25 天（2026-07-06 ~ 2026-07-30）
- 技术栈：SpringBoot 2.7.x、Vue 3、MySQL 8.0、Redis 6.x、Kafka 3.x、Spark 3.x
- 部署方式：Docker Compose（开发/测试），生产环境待定
- 安全要求：JWT 无状态认证、多租户行级隔离、API 限流

### 4.2 用例详述

#### 用例 UC-01：用户登录

| 项目 | 说明 |
|---|---|
| 用例名称 | 用户登录 |
| 参与者 | 管理员、教师、学生、HR |
| 前置条件 | 用户账号已存在且状态正常 |
| 基本流程 | 1. 用户访问登录页<br>2. 输入用户名、密码<br>3. 系统校验账号密码<br>4. 系统返回 JWT Token 与角色信息<br>5. 用户进入对应角色首页 |
| 异常流程 | 用户名或密码错误 → 返回 401 错误提示 |
| 后置条件 | 用户登录态写入前端存储，后续请求携带 Token |

| 基本流程 | 1. 管理员查看采集任务列表<br>2. 新增/编辑/删除任务（来源、URL、Cron 表达式）<br>3. 手动触发或启用定时采集<br>4. 查看采集日志 |

## 5. 非功能需求

### 5.1 性能需求

| 指标 | 目标值 |
|---|---|
| 登录接口响应时间 | ≤ 200ms |
| Dashboard 查询响应时间 | ≤ 1s（无缓存）/ ≤ 200ms（有缓存） |
| 职位列表查询响应时间 | ≤ 500ms |
| 推荐 Top 20 响应时间 | ≤ 2s |
| 支持并发用户数 | ≥ 100（开发/测试环境） |

### 5.2 可用性需求

- 系统 7×24 小时可访问（开发/测试环境除外）。
- 关键业务操作有明确的成功/失败提示。
- 前端页面响应式，适配主流浏览器（Chrome、Edge、Firefox）。

### 5.3 安全性需求

- 用户密码使用 BCrypt 加密存储。
- JWT Token 设置合理过期时间，禁止明文传输。
- 多租户数据按 `tenant_id` 行级隔离。
- 对外 API 使用 OAuth2 + 限流保护。
- 敏感配置（数据库密码、API Secret）不提交到 Git。

### 5.4 可维护性需求

- 后端代码遵循 MVC 分层架构。
- 每个 Java 类不超过 500 行，Vue 文件不超过 300 行。
- 关键节点记录日志（INFO/WARN/ERROR）。
- 接口变更需同步更新接口文档与调用方。

### 5.5 可扩展性需求

- 模块化单体架构，未来可拆分为微服务。
- 分析任务按维度独立，便于新增分析维度。
- 推荐算法先规则后模型，便于逐步升级。

## 7. 业务规则

| 规则编号 | 规则描述 | 适用场景 |
|---|---|---|
| BR-01 | 所有业务表必须包含 `tenant_id` 字段（除明确共享表） | 数据库设计 |
| BR-02 | 用户角色分为 ADMIN、TEACHER、STUDENT、HR | 权限控制 |
| BR-03 | 薪资筛选使用区间交集判断 | 职位查询、推荐匹配 |
| BR-04 | 匹配分 = 技能 40% + 城市 25% + 薪资 20% + 学历 15% | 推荐算法 |
| BR-05 | 投递成功必须同时生成行为记录和推送通知 | 投递流程 |
| BR-06 | 对外 API 每个 apiKey 每分钟最多 60 次 | 限流规则 |
| BR-07 | 报告模板按租户隔离 | 模板管理 |
| BR-08 | 采集任务失败需记录日志，便于排查 | 采集管理 |
