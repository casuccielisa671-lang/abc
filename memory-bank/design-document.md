## 1. 项目背景与目标

### 1.1 背景
当前高校人才培养与就业市场需求之间存在信息不对称问题。学校教学方向、学生技能储备往往滞后于行业实际用人需求。传统的就业指导手段无法实时、精准地反映行业变化趋势。

### 1.2 目标
搭建一套基于大数据技术的职业能力分析平台，实现：
- **全量采集**：多渠道、持续采集行业就业与职位数据。
- **智能分析**：利用大数据算法对职位信息进行多维度统计和分析。
- **自动化报告**：生成各行业的就业形势分析报告，辅助教学改革与管理决策。
- **精准推送**：向学生推送与个人能力匹配的就业岗位信息。
- **对外开放**：提供标准 API，供第三方系统集成调用。

## 3. 功能模块设计

### 3.1 分布式数据采集模块
- **职责**：从多渠道采集职位与就业数据。
- **采集渠道**：
  - **主流招聘平台**：BOSS 直聘、智联招聘等（分布式爬虫，适配反爬策略）。
  - **企业官网**：目标行业头部企业的招聘页（定向采集）。
  - **政府公开数据**：人社局、统计局等发布的就业统计数据（数据对接）。
  - **合作机构数据**：通过 API 接入的第三方数据源。
- **关键能力**：
  - 分布式爬虫调度（基于 Zookeeper 协调）。
  - 反爬策略应对（IP 轮换、UA 伪装、验证码识别）。
  - 数据去重与初筛。
  - 采集任务配置化，支持定时/手动触发。
- **技术选型**：Java + Kafka（消息队列解耦采集与处理链路）。

### 3.2 报告分析系统
- **职责**：对清洗后的职位数据进行多维度分析。
- **分析维度**（初步规划）：
  - 行业维度：行业热度、薪资水平、岗位数量趋势。
  - 技能维度：热门技能词云、技能组合关联分析。
  - 地域维度：城市/区域就业需求分布。
  - 学历与经验维度：不同学历层次的需求占比。
  - 时间趋势：按周/月/季度的岗位变化趋势。
- **技术选型**：Spark（离线批处理）+ Hive（数据仓库）+ HBase（明细查询）。

### 3.3 报告自动生成系统
- **职责**：基于分析结果，自动生成图文并茂的就业分析报告。
- **关键能力**：
  - 报告模板管理（支持多行业、多类型模板）。
  - 数据图表自动渲染（柱状图、折线图、饼图、热力图等）。
  - 报告导出格式：PDF、Word、HTML。
  - 定时生成 + 手动触发生成。
  - 报告归档与历史版本管理。

### 3.4 就业岗位推送系统
- **职责**：基于学生画像与职位标签进行匹配推荐。
- **关键能力**：
  - 学生能力画像构建（专业、技能、成绩、意向城市等）。
  - 职位标签化与向量化。
  - 匹配算法（基于规则 + 协同过滤 + 内容推荐）。
  - 推送渠道：站内通知、邮件、短信。
  - 反馈闭环：学生投递/忽略行为回收，优化推荐模型。

### 3.5 对外公开 API
- **职责**：向第三方系统提供标准化数据接口。
- **接口分类**：
  - 职位查询 API（按条件检索职位数据）。
  - 行业报告 API（获取最新/历史分析报告摘要）。
  - 技能热度 API（查询技能热度排名与趋势）。
  - 统计大盘 API（宏观就业指标数据）。
- **非功能性要求**：鉴权（Token/OAuth2）、限流、日志审计、接口文档（Swagger）。

## 3.5+ ⭐ 差异化亮点模块（核心竞争力）

> 以下 6 个模块是本项目区别于常规大数据就业平台的核心壁垒。

### A. 技能缺口诊断引擎（Skill Gap Analyzer）

- **核心命题**：自动诊断"学校教了什么 vs 市场要了什么"之间的差距。
- **工作流程**：
  1. 院校录入本校课程体系（课程名 + 涵盖技能标签）。
  2. 系统对比市场职位技能需求 → 计算覆盖率矩阵。
  3. 输出"技能缺口清单"：哪些市场热需技能本校课程未覆盖。
- **输出示例**：
  > "贵校《Java程序设计》课程覆盖了 Java SE、MySQL，但市场 78% 的 Java 岗位还要求 Spring Boot、Docker、Redis。建议在课程中增加这 3 个模块（预计可提升学生匹配岗位数 2.3 倍）。"
- **教师端价值**：每条建议附带市场数据证据（岗位数量、薪资溢价），辅助课程改革决策。
- **技术方案**：Spark 离线计算技能共现矩阵 + 课程-技能映射表 + LLM 生成自然语言建议。

### B. 职业能力知识图谱（Competency Knowledge Graph）

- **核心命题**：不只是"Java 热门"，而是"学完 Java 再学 Spring，匹配岗位增加 3 倍"的因果关系。
- **图谱实体**：
  - 技能节点（Java、Spring、Docker、机器学习...）
  - 职位节点（Java开发、大数据工程师、AI工程师...）
  - 行业节点（互联网、金融、制造...）
  - 公司节点（腾讯、阿里、华为...）
  - 关系边：`要求技能`、`前置技能`、`协同技能`、`晋升路径`、`薪资区间`
- **可查询问题**（Cypher）：
  - "从'Java基础'出发，3 跳之内能到达哪些高薪职位？"
  - "Python + SQL 的关联技能是什么？学习后平均薪资涨幅？"
  - "当前市场最稀缺、且本校学生掌握率 < 10% 的技能有哪些？"
- **可视化**：前端使用 D3.js 或 G6（蚂蚁金服图可视化引擎）渲染交互式技能图谱。
- **技术方案**：Neo4j 图数据库 + Spark 批量构建关系边 + G6 前端可视化。

### C. LLM 智能报告引擎

- **核心命题**：传统报告是"把数据填进模板"，智能报告是"像分析师一样写出洞察"。
- **差异化**：
  - 模板方案：固定章节 + 数据填充 + 图表 → 千篇一律。
  - LLM 方案：基于分析数据 + 行业上下文 → **生成自然语言叙事 + 个性化建议 + 行动指南**。
- **报告分层**：
  - **学生版**："你的技能画像中 Docker 和 K8s 缺失，当前市场 65% 的后端岗位要求此技能。建议优先学习 Docker（入门 2 周），匹配岗位数可增加 40%。"
  - **教师版**："贵院计算机专业学生 Spring Boot 掌握率仅 35%，而市场要求率 78%。建议下学年增加 16 学时 Spring Boot 实训。"
  - **校领导版**：宏观趋势 + 对标院校分析 + 战略建议。
- **技术方案**：接入 LLM API（如通义千问/DeepSeek）+ Prompt 工程 + 数据注入模板。

### D. 实时就业市场脉搏（Live Market Pulse）

- **核心命题**：就业市场是动态变化的，不应只看月度报告。
- **功能**：
  - 实时流处理 Kafka Streaming → 每分钟更新"市场热度指数"。
  - 大屏展示：城市热力变化、技能需求涨跌排行、新增岗位动态。
  - 异常检测：某行业岗位数突然暴跌 30% → 告警通知。
  - "今日就业指数"：像股市指数一样，每天发布综合就业热度数值。
- **技术方案**：Spark Streaming / Flink + Redis 实时计数器 + WebSocket 推送前端。

### E. NLP 简历-JD 深度匹配

- **核心命题**：规则打分只能做"Java 匹配 Java"，但"Spring Boot 微服务开发"和"Spring Cloud 分布式"是高度相关的——需要语义理解。
- **方案**：
  - 使用中文预训练模型（如 text2vec / BGE）将 JD 和学生简历/技能向量化。
  - 余弦相似度计算语义匹配度。
  - 输出匹配度 + **差距分析**：高亮"你缺什么"。
- **技术方案**：Python Flask 服务 + Sentence-Transformers 模型 + Milvus/FAISS 向量检索。

### F. 就业趋势预测模型

- **核心命题**：不只是"现状分析"，而是"未来预判"。
- **预测维度**：
  - 某行业 3/6 个月后的岗位数量预测（时序模型 ARIMA/Prophet）。
  - 某技能的需求涨跌趋势（基于历史热度曲线）。
  - 某城市薪资走势预测。
- **应用场景**：
  - 学生选课参考："预测 AI 工程师岗位 6 个月后增长 25%，建议现在投入学习。"
  - 学校专业设置："大数据专业未来 2 年岗位供给 > 需求，建议控制招生规模。"
- **技术方案**：Prophet/FlinkML + Spark 历史数据训练 + 定时更新模型。

## 4. 用户角色与多租户设计

### 4.1 用户角色

系统面向四类角色，不同角色拥有独立的功能视图和权限边界：

| 角色       | 核心功能                                                                 | 典型场景                                       |
| ---------- | ------------------------------------------------------------------------ | ---------------------------------------------- |
| **学生**   | 查看就业报告、接收岗位推送、管理个人画像、收藏/投递职位                   | 了解行业趋势，匹配合适岗位                     |
| **教师/辅导员** | 查看所辖班级学生就业动态、查看教学建议报告、批量导出学生就业数据         | 掌握学生就业情况，调整教学内容                 |
| **管理员** | 系统配置、租户管理、数据采集调度、报告模板管理、用户管理、日志审计       | 平台日常运维与配置管理                         |
| **企业 HR** | 发布职位、查看人才画像（脱敏后）、接收学生投递                           | 企业发布岗位，对接院校人才                     |

### 4.2 SaaS 多租户架构

- **模式**：单实例多租户，同一套系统服务多所院校。
- **数据隔离**：核心业务表通过 `tenant_id` 字段实现行级隔离；分析结果数据按租户维度独立存储与计算。
- **租户管理**：管理员可为每个租户（院校）独立配置：
  - 可用的采集数据源。
  - 报告模板与推送策略。
  - 用户数量上限与功能开关。
- **鉴权**：统一登录入口，JWT Token 携带租户标识，后端拦截器校验租户权限。

## 6. 数据流设计（升级版）

```
                            ┌──→ [Neo4j 知识图谱] ←──┐
                            │                         │
多源数据 → [采集] → Kafka → [清洗] → HDFS/Hive → [Spark分析] → MySQL
                │                              │                    │
                │              ┌───────────────┘                    ↓
                │              ↓                              [结果表 + 缓存]
                │        [Elasticsearch]                            │
                │         (全文检索)                                ↓
                │                                        ┌──────────────────┐
                └──→ [实时流处理] ──→ Redis计数器 ──→   │ 实时市场脉搏大屏  │
                                                        └──────────────────┘
                                                                  │
                                          ┌───────────────────────┘
                                          ↓
                              ┌───────────────────────┐
                              │    ⭐ AI 智能引擎层     │
                              │                        │
                              │  LLM → 智能报告/教学建议│
                              │  NLP → JD-简历语义匹配  │
                              │  Prophet → 趋势预测     │
                              │  知识图谱 → 关联推理    │
                              └───────┬───────────────┘
                                      │
                          ┌───────────┼───────────┐
                          ↓           ↓           ↓
                     学生端报告   教师端建议   校领导决策
                    (个人洞察)   (课程改革)   (战略规划)
```

1. **采集**：爬虫/API 接入 → 原始数据写入 Kafka。
2. **清洗**：消费 Kafka 消息 → 去重、标准化 → 写入 HDFS（原始层）和 Hive（结构化层）。
3. **分析**：Spark 定时任务读取 Hive 数据 → 多维度计算 → 结果写入 MySQL。
4. **应用**：Web 后台读取 MySQL 结果 → 报告渲染生成；推荐引擎读取学生画像 + 职位标签 → 匹配推送。

## 7. 核心设计原则

1. **解耦**：采集、清洗、分析、应用各环节通过 Kafka 解耦，独立扩展。
2. **分层存储**：原始数据（HDFS）→ 清洗数据（Hive）→ 结果数据（MySQL/Redis），各层职责清晰。
3. **模块化**：每个功能模块独立部署、独立维护，通过 API 互相调用。
4. **可扩展**：采集源、分析维度、报告模板均支持配置化新增，无需改动核心代码。
5. **可靠性**：关键链路（采集→Kafka→存储）具备重试机制和监控告警。
