二、启动步骤（按顺序）

第 1 步：起基础设施

cd abc
docker-compose up -d
docker-compose ps        # 确认 5 个容器都是 healthy/running

起来的是：MySQL(3306)、Redis(6379)、Zookeeper(2181)、Kafka(9092)、Nginx(80)。

数据库初始化是自动的：compose 把 init.sql 挂载到了 MySQL 的 /docker-entrypoint-initdb.d/，首次启动容器时自动建 14 张表 + 种子数据（账号 admin / admin123）。注意"首次"——如果之前起过容器又改了 SQL，要 docker-compose down -v 删掉数据卷重来，或者手动用 Navicat 执行 init.sql。

第 2 步：配置后端连接参数

所有配置在 occupation-web/src/main/resources/application.yml，环境相关项都支持环境变量覆盖。用 Docker 数据库时必须处理密码问题，三选一：

- 方式 A（推荐，不动代码）：IDEA 运行配置里加环境变量 DB_PASSWORD=root
- 方式 B：命令行 set DB_PASSWORD=root && mvn spring-boot:run（PowerShell 是 $env:DB_PASSWORD='root'）
- 方式 C：直接改 yml 里的默认值（不建议提交个人密码，这也是实训规范要求）

yml 里的功能开关：app.scheduler.enabled（定时任务，调试时想安静可关）、xxl.job.enabled（默认 false，别开，开了要单独部署 xxl-job-admin）、app.kafka.auto-create-topics（默认 false，Kafka 的 compose 配置已开自动建 Topic，所以不用管）。

第 3 步：起后端

cd abc
mvn clean compile -DskipTests     # 首次拉依赖会比较久
cd occupation-web
mvn spring-boot:run

IDEA 里等价操作：打开 abc 根目录 → 等 Maven 导入完 → 运行 occupation-web 模块的 Application.java（记得在 Run Configuration 加 DB_PASSWORD=root）。

验证：浏览器开 http://localhost:8080/api/health，返回 {"code":200,...,"data":"ok"} 即成功。

第 4 步：起前端

cd abc/occupation-web-ui
npm install
npm run dev

访问 http://localhost:5173。Vite 已配置把 /api 代理到 8080，开发阶段不需要 Nginx（Nginx 是给生产部署演示用的：前端 npm run build 后把产物放进容器，走 80 端口访问）。

第 5 步：灌演示数据（让 Dashboard 有图）

用 Postman/Apifox 按顺序调（先登录拿 token，后续请求带 Authorization: Bearer {token}）：

1. POST /api/auth/login          body: {"username":"admin","password":"admin123","tenantId":1}
2. POST /api/admin/crawler/task/mock     ← 启动模拟爬虫（20 条职位 → Kafka → 清洗入 job_detail）
3. 等 10 秒左右，POST /api/analysis/rebuild   ← 手动触发统计（不用等凌晨 2 点的定时任务）
4. GET  /api/analysis/dashboard?tenantId=1    ← 应返回 5 个维度的数据

这条链路通了，说明 采集→Kafka→清洗→统计→查询 整个管道都是好的。

三、怎么调试

后端调试（IDEA）

- 断点调试：Debug 模式启动 Application，在 Controller/Service 打断点，用 Postman 发请求即可命中。整个项目是单体应用，8 个模块跑在一个 JVM 里，跨模块调用（如 recommend 调 analysis 的 Service）也能直接单步跟进去，这是模块化单体对新手最友好的地方。
- 看 SQL：yml 里已开 log-impl: StdOutImpl，控制台会打印每条 SQL——验证多租户插件时特别有用：登录后发请求，看 SQL 的 WHERE 里是否自动多了 tenant_id = 1。
- 调试带认证的接口：每次都要带 token 很烦，两个办法：① Postman 环境变量存 token；② 临时把要调的路径加进 JwtAuthenticationFilter 的 WHITE_LIST（调完记得删，别提交）。
- 调定时任务：不要傻等 cron 时间。清洗补偿、统计重算都有手动入口（POST /api/analysis/rebuild），或者在 IDEA 里直接写个测试调 analysisScheduler.rebuildAnalysis()。

中间件排查

# Kafka：看消息有没有进来（验证爬虫→Kafka 这一段）
docker exec -it occupation-kafka kafka-console-consumer --bootstrap-server localhost:9092 --topic raw-job-data --from-beginning

# Redis：看缓存/开放API Token/限流计数
docker exec -it occupation-redis redis-cli
> keys open-api:*

# MySQL：直接查表验证链路每一环
docker exec -it occupation-mysql mysql -uroot -proot occupation
> SELECT COUNT(*), status FROM raw_job_data GROUP BY status;   -- 原始层
> SELECT COUNT(*) FROM job_detail;                              -- 清洗层
> SELECT dimension, COUNT(*) FROM analysis_result GROUP BY dimension;  -- 汇总层

数据管道出问题就按这三层逐段定位：raw_job_data 没数据 → 查爬虫/Kafka；有 raw 没 job_detail → 查清洗监听器日志；有明细没结果 → 查统计任务。
